# Determinism and seeds

Data generation is intentionally reproducible. The same plan and seed
produce byte-identical data, on any cluster, at any scale. This makes 
the output reusable in test fixtures, regression baselines, and error 
reproductions.

This page explains the seed model and how to vary it when you need to 
generate unique data from a plan.

## The seed model

`DataGenPlan.seed` is a single global seed for the whole plan that is 
**propagated to each table**. Any `TableSpec` whose own `seed` is `None` 
(the default case) has its seed set to `plan.seed + i`, where `i` is the
table's index in the plan. Tables are independently seeded but still fully 
determined by the plan's global seed. 

Within a table, each cell's value is derived deterministically from the 
table seed and the row index, so generation is reproducible even though it
is distributed across Spark partitions.

### Setting a seed value

Pass a `seed` when creating a `DataGenPlan` or `TableSpec`:

```python
DataGenPlan(seed=42, tables=[a, b, c])
# a.seed -> 42, b.seed -> 43, c.seed -> 44  (when each table.seed is None)
```

Set a `TableSpec.seed` explicitly only when generating a single table
with `generate_table` outside a plan.

### Default seed value

`DataGenPlan.seed` uses a default seed value of `42`. Independent callers 
that do not specify a `seed` will produce identical data. Set a non-default
seed to control variability across runs.

## Reproducible UUIDs and patterns

UUIDs, pattern columns, values columns, and range columns are still 
seed-deterministic:

- `UUIDColumn` derives each UUID from `(table seed, row index)`. The
  same seed and row always yield the same UUID.
- `PatternColumn`, `ValuesColumn`, `RangeColumn`, etc. all draw from
  the same seeded stream.

## Correlated columns: `seed_from`

To make two columns agree (e.g. the same `device_id` always maps to
the same `country`) set `seed_from` on the dependent column to the
name of the column it should track. Equal values in the source column
produce equal values in the dependent one. See [correlated-columns.md](../relationships/correlated-columns.md).

## Varying data at runtime

A common question: *"How do I get fresh, different data on every run?"*
Seed values should be generated by the caller for each generation run. To
randomize across runs, pass a randomly-generated seed value:

```python
import random

plan = DataGenPlan(seed=random.randint(0, 2**63 - 1), tables=[...])
```

For a plan loaded from a file, override the seed at load time:

```python
import random, yaml
from dbldatagen.core import DataGenPlan

raw = yaml.safe_load(open("plan.yml").read())
raw["seed"] = random.randint(0, 2**63 - 1)   # fresh each run
plan = DataGenPlan.model_validate(raw)
```

This keeps specs deterministic by default and makes "different
data each run" an explicit, opt-in caller choice — see [loading-plans.md](../loading-plans.md).

## Reproducing a specific dataset later

Because output is fully determined by the plan and seed, the exact dataset
can be regenerated across runs. Log the seed you used (especially if you 
randomized it), and you can reproduce a run byte-for-byte by passing the 
same value in a subsequent run.
